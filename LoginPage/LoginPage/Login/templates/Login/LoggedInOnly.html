How Web Sessions Work

Web applications are naturally stateless. This means that every time your browser sends a request to a server, the server treats it as if it has never seen you before.

Example:
  • You open a website
  • You log in
  • You click another page
Without sessions, the server would forget who you are after every page request.

The Problem

Imagine a website with a login system.

If the server forgets every user after each request:

  • You would have to log in again on every page
  • The server would not know who you are
  • Your cart, settings, and preferences would disappear

To solve this problem, web applications use sessions.

What is a Session?

A session is a temporary way for a website to remember who a user is between requests.

Instead of storing all user information in the browser, the server creates a unique Session ID.

Think of the Session ID like a ticket number.
  • The server stores information about the user
  • The browser stores the Session ID
  • Each request sends the Session ID back to the server

How Sessions Work (Step by Step)

  1. User logs in with a username and password.
  2. The server verifies the login credentials.
  3. If correct, the server generates a unique Session ID.
  4. The server stores session data such as:
    • User ID
    • Login status
    • Permissions
  5. The server sends the Session ID to the browser.
  6. The browser stores the Session ID (usually inside a cookie).
  7. Every time the browser requests another page, the Session ID is sent with the request.
  8. The server reads the Session ID and retrieves the stored session data.

Analogy: Cloakroom Ticket

  • You give your coat to the attendant.
  • You receive a ticket number.
  • The attendant stores your coat.
  • You return with your ticket.
  • The attendant retrieves your coat.
Your browser holds the ticket. The server holds the coat.

Example Session Data

A session stored on the server might look like this: 

Session ID: A92F8C1D

Stored Data
-----------
username = hayden
logged_in = true
role = admin

Notice that the browser only stores the Session ID, not the actual user data.

Why Sessions Are Important

  • Keep users logged in
  • Store shopping carts
  • Remember user preferences
  • Manage authentication

Security Considerations

Because the Session ID identifies a user, it must be protected.

  • Session hijacking
  • Session fixation
  • Session theft

Common protections include:

  • Secure cookies
  • HTTPS encryption
  • Session expiration
  • Regenerating session IDs after login

Summary

  • Websites are stateless by default.
  • Sessions allow servers to remember users between requests.
  • A Session ID links the browser to data stored on the server.
  • Sessions power login systems, carts, and personalised experiences.
