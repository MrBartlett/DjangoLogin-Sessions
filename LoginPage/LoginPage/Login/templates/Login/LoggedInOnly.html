How Web Sessions Work

Web applications are naturally stateless. Every request is treated independently — the server does not remember previous interactions.

Without sessions:
  • You log in
  • You click another page
  • The server forgets who you are

The Core Problem

Modern web applications require state:

  • Authentication (Who are you?)
  • Authorisation (What can you access?)
  • Persistence (What have you done?)

Without a solution, every page would require re-authentication — which is clearly impractical.

What is a Session?

A session is a server-side mechanism used to track a user across multiple requests.

A session works by linking:
  • Client (browser) → stores Session ID
  • Server → stores user data

The Session ID acts as a secure reference — not the data itself.

Session Lifecycle (Step-by-Step)

  1. User submits login credentials
  2. Server validates credentials (Authentication)
  3. Server generates a unique Session ID
  4. Server stores session data (user, role, permissions)
  5. Session ID is sent to browser (via cookie)
  6. Browser sends Session ID with every request
  7. Server retrieves session data → grants access

Analogy: Cloakroom System

  • You hand over your coat (data)
  • You receive a ticket (Session ID)
  • The cloakroom stores your coat (server)
  • You present your ticket to retrieve it
Key Idea: The browser never stores the coat — only the ticket.

Example Session Data (Server-Side)

Session ID: A92F8C1D

username = hayden
logged_in = true
role = admin

The browser only holds the Session ID — reducing exposure of sensitive data.

Security Considerations

Sessions are a major attack target because they represent authenticated identity.
  • Session Hijacking – stealing a valid session ID
  • Session Fixation – forcing a known session ID
  • Session Theft – via XSS or insecure transport

Defensive Strategies:

  • HTTPS (prevents interception)
  • Secure + HttpOnly cookies
  • Session expiry / timeout
  • Regenerate session ID after login

Summary

  • HTTP is stateless → sessions introduce state
  • Session ID links browser ↔ server data
  • Used for authentication and authorisation
  • Must be secured to prevent account compromise
